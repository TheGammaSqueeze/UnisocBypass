# Module loading on stock kernel + GPU voltage patch attempt

This documents the path we took after `FOUND_IT.md` to actually apply
the slot-4 grade index patch via a custom kernel module, what worked,
and what stopped us.

## sig_enforce kernel patch (works, persistent)

- Stock kernel has `CONFIG_MODULE_SIG_FORCE=y` and the kernel signs every
  vendor module with an auto-generated key whose private half lives only
  on the vendor's build host (`dell@dell-PowerEdge-R730XD`). PKCS7 inspection
  of `mali_gondul.ko` confirms `commonName="Build time autogenerated kernel key"`,
  serial `93AA4A57CE31ADF9`, RSA-4096 + SHA-512.
- Cannot recover the private key. Can't sign new modules.
- **Patched the kernel image instead.** The `sig_enforce` variable is at
  kernel virtual address `0xffffffc012368350` (file offset `0x22e8350` of
  the raw Image). One-byte flip from `0x01` to `0x00` disables signature
  enforcement.
- Kernel binary is in `boot.img`. Unpacked with `android_boot_image_editor`,
  byte-flipped, repacked, flashed to `mmcblk0p45` and `mmcblk0p46`. Boots
  cleanly. `/sys/module/module/parameters/sig_enforce` reports `N`. Stock
  backup at `/mnt/c/vita/boot_b_stock_backup.img`. Patched image at
  `/mnt/c/vita/boot_oc_sig_disabled.img.signed`.

## Out-of-tree module build chain

- Cloned `marohinmark/kernel_ums512_5.4` (closest public 5.4 UMS512 source)
  to `/work/ums512_kernel`.
- Edited `Makefile`: `SUBLEVEL=161` (matches device).
- `setlocalversion` overridden to print empty string (otherwise `+` suffix).
- Started with the device's actual `/proc/config.gz` as `.config`. This
  has CONFIG_LTO_NONE=y by default, and CONFIG_MODVERSIONS=y.
- Re-enabled CFI in our build: CONFIG_LTO_CLANG=y, CONFIG_CFI_CLANG=y,
  CONFIG_CFI_CLANG_SHADOW=y. Required by the running kernel - without CFI,
  `module_exit` callbacks panic on `Kernel panic - not syncing: CFI failure
  (target: vgv_exit+0x0/0xff8)`.
- Removed `-plugin-opt=-warn-stack-size=$(CONFIG_FRAME_WARN)` from the
  top Makefile (LLD 18 doesn't accept it; vendor used LLD 12).
- Build with `LLVM=1` and the NDK r27c clang/lld toolchain.

A "hello world" module loaded cleanly. vermagic must match
`5.4.161 SMP preempt mod_unload modversions aarch64`. The MODVERSIONS
"no symbol version for module_layout" warning is harmless once
sig_enforce is off.

## GPU voltage-grade patch module

Source: `vita_gpu_voltage.c` - on init, ioremaps GPU syscon at
`0x60110000`, does `regmap_update_bits` on register `0x58`
(core_index4_map), changing bits 14-16 from grade 2 to grade 3.

```
VGV: before 0x308005    <-- slot 4 reg, grade 2 (800 mV)
VGV: after  0x30c005    <-- grade 3 (would be 850 mV per top-DVFS-CTRL grade table)
```

The write succeeds and persists across an immediate read-back. So the
direct patch DOES land in hardware.

## Why the patch doesn't break the 800 mV ceiling

Two independent obstacles:

1. **Mali driver actively rewrites core_indexN_map.** Loaded a separate
   read-only check module after pinning GPU to 900 MHz; it observed
   `core_index4_map = 0x308005, grade = 2` - back to stock value. Our patch
   was undone by the time GPU reached top OPP. The Mali driver's
   `mali_clock_on` path does multiple `regmap_update_bits_base` writes
   through a 9-entry array stored at `mali_gondul.ko .data + 0x1d80`; one
   of those entries writes core_index4_map back to grade 2.
2. **Reading GPU syscon when GPU is power-gated raises async SError.**
   Multiple device crashes during testing trace to
   `SError Interrupt on CPU?, code 0xbe000011, comm: awk/insmod, PC=
   regmap_mmio_read+0x70 / vprintk_func+0x28`. The async-SError pattern
   means a deferred bus error from a `readl(GPU_SYSCON+offset)` arrives
   later when the kernel does a printk or other unrelated work. Any
   userspace or kernel-module-side read of the GPU regmap from outside
   Mali's powered context is unsafe.

## Paths forward (NOT pursued in this session)

To actually break the 800 mV ceiling we'd need any of:

A. **Patch `mali_gondul.ko` binary directly.** Identify the offset in
   `mali_clock_on` where grade 2 is written for slot 4 and change the
   immediate. The .ko is signed; with sig_enforce off our patched copy
   loads. Replace it in `/vendor_dlkm/lib/modules/`. AVB is already off
   so the partition rewrite goes through.

B. **kprobe `regmap_update_bits_base` with a per-call filter** that
   detects writes to GPU syscon offset 0x58 and rewrites the value to
   grade 3. Run inside our existing kernel module via `kprobe_register`.
   Only takes effect AFTER mali_clock_on returns, so requires the kprobe
   to have a post handler.

C. **Replace Mali's gpu_dvfs_ctx.core_index4_map regmap pointer** with
   one that has a custom write callback that ignores grade-2 writes.
   Significantly more invasive.

D. **Power on the GPU subsystem manually** (via runtime PM) before
   reading/writing the syscon, to avoid the async SError. Then re-write
   slot 4 in a workqueue that runs after every GPU OPP transition.

The amount of work for any of A-D exceeds the value of pushing the GPU
above 800 mV given that exp 020/025 already showed 900 MHz runs stably
at the stock 800 mV rail (16.26 FPS, +17% vs stock). 950/1000 MHz with
forced 850 mV via debugfs (exp 028/029) gave zero FPS gain because the
benchmark is shader-saturated, and made both-mode worse thermally.

## Practical conclusion

The module loading infrastructure is now usable and committed
(boot.img patched, sig_enforce off, OOT module build chain proven). The
voltage-grade patch as currently written succeeds on the register but
is undone by Mali's clock-on path. Finishing the work means binary-
patching the signed mali_gondul.ko, which is the cleanest of the four
options. Recommended next-session direction.
